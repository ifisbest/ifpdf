# ifpdf 产品需求文档
## ——面向人文社科学术研究的扫描版 PDF OCR 与结构化转换工具

---

## 1. 背景与问题陈述

### 1.1 用户场景

人文社科研究者的核心工作流：
1. 从图书馆、数据库、Anna's Archive 等渠道获取扫描版学术专著 PDF
2. 需要细读、标注、引用这些文本
3.  increasingly 使用 AI（Claude、ChatGPT、Cursor）辅助阅读、比较、综述

**核心痛点**：扫描版 PDF 是纯图片，AI 完全无法读取。研究者陷入两难——要么逐字手打关键段落（不可行），要么放弃 AI 辅助（落后）。

### 1.2 现有工具的局限性

| 工具/方案 | 功能 | 核心缺陷 |
|----------|------|---------|
| **Adobe Acrobat Pro** | OCR + 导出 | ① 订阅制付费；② GUI 操作，无法批量自动化；③ 导出 Word/Markdown 格式混乱，页码信息丢失；④ 中文竖排古籍识别率极低 |
| **ABBYY FineReader** | 专业 OCR | ① 付费软件，价格高昂；② 无 CLI，无法集成到工作流；③ 输出格式不友好 |
| **Tesseract CLI** | 开源 OCR | ① 纯命令行，无 PDF 处理管道；② 需要手动拆页、调参、后处理；③ 无中文分词优化；④ 输出纯文本，无结构、无页码 |
| **Calibre + 插件** | 电子书管理 | ① 不处理扫描版 PDF；② 转换目标是 epub/mobi，非学术引用场景 |
| **在线 OCR 服务** (iLovePDF, Smallpdf) | 上传转换 | ① **隐私风险**：敏感学术资料上传第三方服务器；② 无法自动化批量处理；③ 输出质量不可控；④ 中文识别精度差 |
| **Marker / Nougat (Meta)** | AI 模型驱动的 PDF→Markdown | ① 需要 PyTorch + 2GB+ 模型下载；② 针对英文学术论文优化，中文专著效果极差；③ 首次运行极慢；④ 不处理扫描版 |
| **Mac 自带预览「复制文本」** | 快速提取 | ① 仅对有文字层的 PDF 有效；② 扫描版完全无法使用；③ 无批量能力 |

**总结**：市场上**没有一款工具**同时满足：① 扫描版中文 PDF OCR、② 保留精确页码、③ CLI 批量自动化、④ 输出 AI 友好格式。

## 2. 产品定位

**一句话**：让几百页的扫描版学术专著变成 AI 能精确引用到页码的结构化 Markdown。

**目标用户**：
- 人文社科博士/研究者，个人文献库以扫描版 PDF 为主
- 需要 AI 辅助阅读、比较、综述大量文献
- 对学术引用精度有要求（必须能定位到具体页码）

**差异化定位**：

| 维度 | 现有工具 | ifpdf |
|-----|---------|-------|
| 目标文档 | 英文论文 / 有文字层的 PDF | **扫描版中文学术专著** |
| 输出目标 | 人类阅读 / 存档 | **AI 理解 + 精确引用** |
| 页码处理 | 无 / 物理页码 | **正文页码映射 + 引用标记** |
| 元数据 | 无 / 手动 | **自动从文件名提取 + 交互补全** |
| 批量处理 | GUI 逐个操作 | **CLI 批量 + 多进程并行** |
| 自动化 | 无 | **管道友好，可集成到文献库工作流** |

## 3. 功能规格

### 3.1 核心功能

#### 3.1.1 智能 OCR 管道

```
PDF 输入
  ↓
[检测] 有文字层？→ 是：直接提取
              → 否：OCR 流程
  ↓
[渲染] pdf2image → 高分辨率 PNG (300 DPI)
  ↓
[预处理] Pillow：二值化、降噪、对比度增强
  ↓
[识别] Tesseract (chi_sim) / PaddleOCR (可选)
  ↓
[后处理] 合并断行、修正常见 OCR 错误、识别章节标题
  ↓
Markdown 输出
```

**关键设计**：
- 自动检测扫描版 vs 文字层 PDF，无需用户判断
- 多进程并行（`--workers N`），600 页书从 50 分钟 → 10-15 分钟
- 进度条实时显示（"正在 OCR 第 247/637 页..."）

#### 3.1.2 页码对齐系统

**问题**：扫描版 PDF 的物理页 ≠ 正文页码
- 封面、版权页、前言、目录 = PDF 前 N 页，无正文页码
- 正文从 PDF 第 P 页开始，对应实际 "第 1 页"

**解决方案**：
```bash
ifpdf ocr book.pdf --content-starts-at 25
# PDF 第 25 页 = 正文第 1 页
```

输出格式：
```markdown
<!-- 第 1 页 -->
第一章 导论

中国古代小说的发展历程源远流长...

<!-- 第 2 页 -->
对于"小说"这一概念，历代学者的理解并不一致...
```

**引用场景**：
- 研究者问 AI："刘勇强如何看待《红楼梦》的叙事结构？"
- AI 回答："根据刘勇强《中国古代小说史叙论》**第 87 页**的观点..."

#### 3.1.3 元数据自动提取

**文件名解析**（从 Anna's Archive 等渠道下载的常见命名）：
```
中国古代小说史叙论-_刘勇强著_2007_--_北京市：北京大学出版社_--_9787301122303
```

自动提取：
- 书名：中国古代小说史叙论
- 作者：刘勇强
- 出版年：2007
- 出版社：北京大学出版社
- ISBN：9787301122303

**交互式补全**（缺失字段自动询问）：
```bash
$ ifpdf ocr book.pdf --interactive
书名: 中国古代小说史叙论 [回车确认]
作者: 刘勇强 [回车确认]
出版社: [未检测到，请输入] 北京大学出版社
出版年: 2007 [回车确认]
ISBN: 9787301122303 [回车确认]
正文起始页（PDF页号）: 25
```

输出头部：
```markdown
# 中国古代小说史叙论

**作者**: 刘勇强
**出版社**: 北京大学出版社
**出版年**: 2007
**ISBN**: 9787301122303
**总页数**: 637
**OCR日期**: 2026-04-17
**正文起始页**: 25 (PDF页)

---
```

#### 3.1.4 智能分块

**问题**：LLM 有上下文限制（Claude 200K tokens ≈ 15 万字），一本 30 万字的书无法一次性塞入。

**解决方案**：
```bash
ifpdf ocr book.pdf --content-starts-at 25 --chunk-size 4000
```

输出：
```markdown
<!-- === CHUNK 1/15 (第 1-45 页) === -->
[内容]

<!-- === CHUNK 2/15 (第 46-91 页) === -->
[内容]
```

每块带页码范围，AI 回答时可以精确定位引用来源。

#### 3.1.5 批量处理

```bash
# 批量转一个文件夹
ifpdf batch ./books/ -o ./output/

# 只处理小于 50 页的（自动识别为论文）
ifpdf batch ./papers/ --max-pages 50

# 多进程并行处理多个文件
ifpdf batch ./books/ --workers 4
```

**批量规则**：
- 自动检测每本 PDF 是文字层还是扫描版
- 文字层直接提取（秒级），扫描版走 OCR（分钟级）
- 输出按原文件名命名的 `.md`
- 失败文件记录到 `ifpdf_errors.log`

## 4. 与 AI 工作流的配合

### 4.1 文献库流水线

```
[扫描版 PDF 书籍]
        ↓
   ifpdf ocr
        ↓
[结构化 Markdown (.md)]
        ↓
放入 ~/文献库/
        ↓
OpenClaw / Claude Code 读取
        ↓
AI 问答 + 精确引用
```

### 4.2 AI 辅助阅读的增强

**转换前**（直接上传 PDF）：
- AI："抱歉，我无法读取这个 PDF 的内容。"

**转换后**（上传 Markdown）：
- 研究者："请总结刘勇强对《红楼梦》叙事结构的分析"
- AI："根据刘勇强《中国古代小说史叙论》（北京大学出版社 2007）**第 234-238 页**的论述，作者从三个层面分析了《红楼梦》的叙事结构：..."

### 4.3 高级用法：RAG 管道

```bash
# 1. 批量转换整个文献库
ifpdf batch ~/文献库/pdfs/ -o ~/文献库/markdown/

# 2. 按 chunk 分块（适合向量数据库）
for f in ~/文献库/markdown/*.md; do
    ifpdf chunk "$f" --chunk-size 2000 -o ~/文献库/chunks/
done

# 3. 导入向量数据库（如 Chroma、Pinecone）
# 4. AI 问答时做语义检索 + 精确引用
```

## 5. 技术架构

```
ifpdf/
├── src/ifpdf/
│   ├── __init__.py
│   ├── cli.py              # Typer CLI（convert, ocr, batch, info, chunk）
│   ├── extractor.py        # 文字层 PDF 提取（PyMuPDF）
│   ├── ocr_engine.py       # OCR 引擎封装（Tesseract / PaddleOCR）
│   ├── preprocessor.py     # 图像预处理（二值化、降噪）
│   ├── layout.py           # 布局分析（标题/正文/页眉页脚识别）
│   ├── formatter.py        # Markdown 格式化输出
│   ├── chunker.py          # Token 感知分块（tiktoken）
│   ├── metadata.py         # 文件名解析 + 交互式补全
│   ├── pagemap.py          # 页码映射（PDF页 → 正文页）
│   └── utils.py            # 文件操作、剪贴板、进度显示
├── tests/
├── pyproject.toml
└── README.md
```

### 5.1 技术选型

| 组件 | 选择 | 理由 |
|-----|------|------|
| 语言 | Python 3.9+ | PDF/OCR 生态最成熟 |
| PDF 渲染 | pdf2image + poppler | 高质量图片输出 |
| OCR 引擎（默认） | Tesseract (chi_sim) | 开源、离线、中文支持 |
| OCR 引擎（可选） | PaddleOCR | 中文快 2-3 倍、精度更高 |
| 图像处理 | Pillow | 标准库、预处理必需 |
| CLI 框架 | typer | 类型安全、自动生成 help |
| TUI | rich | 进度条、表格、彩色输出 |
| Token 计算 | tiktoken | 与 OpenAI 模型对齐 |
| 并行处理 | multiprocessing | 利用多核加速 OCR |
| 打包 | setuptools | 标准方案、pip 可安装 |
